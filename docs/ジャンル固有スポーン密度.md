# ジャンル固有スポーン密度

## 概要

各ジャンルに固有の障害物（ハザード）出現密度を設定する機能。これにより、STG では障害物が密に、Idle では障害物が疎に出現するなど、ジャンルの雰囲気に合わせたゲームバランスを実現する。

- Issue: #118
- ブランチ: `fix/118-per-genre-spawn-density`

## 背景・問題定義

### 変更前

障害物の出現密度は全ジャンルで共通の `HAZARD_SPAWN` 定数（`src/data/config/spawn.json`）が適用されていた。

```jsonc
// spawn.json
{
  "baseInterval": 2400,    // ms
  "minInterval": 800,      // ms
  "decayRate": 0.00015     // 距離による密度増加率
}
```

このため、障害物を dense にしたい STG と sparse にしたい Idle とで同じ間隔で障害物が発生し、ジャンルの違いがスポーンパターンから伝わらないという課題があった。

### 変更後

各ジャンルの JSON 定義ファイル（`src/data/genres/*.json`）に `spawnDensity` プロパティを追加し、エンジンサイドでその値を読み込んで使用する。

## 設計

### spawnDensity の構造

```typescript
interface SpawnDensity {
  baseInterval: number  // 初期出現間隔 (ms)
  minInterval: number   // 最短出現間隔 (ms)
  decayRate: number     // 距離による密度増加係数
}
```

### 出現間隔の計算式

```
interval = baseInterval × exp(-decayRate × distance)
finalInterval = max(minInterval, interval)
```

距離が長いほど `interval` が短くなり、障害物密度が高くなる。ただし `minInterval` 以下にはならない。

### データフロー

```
src/data/genres/{genre}.json
    │
    │  GAME_CONFIG.genres.genres[] via ConfigLoader
    ▼
src/genres/index.ts
    │
    ├── ① TS プラグイン registered → mergeSpawnDensity() via GameRegistry
    │   (spawnDensity を TS プラグインインスタンスにはめ込む)
    │
    └── ② TS プラグینなし → JSONGenrePlugin 生成
        (コンストラクタに対して spawnDensity を渡す)
    ▼
src/engine/GameRegistry (Map<GenreId, GenrePlugin>)
    │
    ▼
src/game/sideScroller.ts::_getSpawnParams()
    │
    │  GameRegistry.getGenre(currentGenreId).spawnDensity を取得
    │  undefined の場合は HAZARD_SPAWN にフォールバック
    ▼
  障害物スポーン処理
```

### 重要な設計決定

#### 1. JSON が唯一の truth

`spawnDensity` は JSON 定義ファイルが唯一の真相源（single source of truth）となる。TS プラグイン側で `spawnDensity` をハードコードせず、すべて JSON から読み込む。

理由:
- JSON のみ編集すればゲームバランス調整が可能（再コンパイル不要）
- CLAUDE.md の「JSON 駆動設計」方針に準拠
- TS プラグインの責務を「ロジック実装」に集中できる

#### 2. mergeSpawnDensity の型キャスト

`GameRegistry.mergeSpawnDensity()` 内で `as unknown as Record<string, unknown>` カストを使用している。これは `GenrePlugin` インターフェースの `readonly spawnDensity?` に対して値を設定するため。

安全な理由:
- 起動時に 1 回だけ呼ばれる（イミュータブルな初期化）
- `games/index.ts` が呼び出す前に `def.spawnDensity` の存在をチェック
- `GameRegistry` 内で閉じており、外部からアクセスできない
- Vite の `import.meta.glob` で動的に読み込んだインスタンスである

#### 3. 後方互換性

`spawnDensity` が `undefined` の場合はグローバル `HAZARD_SPAWN` にフォールバックする。既存のジャンル JSON（bullet_hell, horror, stealth_action など 15ジャンル）は変更されないため、既存の挙動が壊れない。

## ジャンルごとの値

| ジャンル | baseInterval | minInterval | decayRate | イントレンション |
|---|---|---|---|---|
| stg | 1200 | 400 | 0.0003 | 密な障害物 = 密な射击ターゲット |
| arena | 1000 | 350 | 0.00025 | 超密な多敵展開 |
| bullet_hell | 800 | 250 | 0.0002 | 超密な弾幕 |
| survival | 1400 | 500 | 0.00025 | 持続的な脅威 |
| idle | 4000 | 2000 | 0.00005 | 緩やかな出現 |
| runner | 1800 | 600 | 0.0001 | 標準的ランナー |

## テスト戦略

### ユニットテスト (`tests/unit/engine/GenrePluginSpawnDensity.test.ts`)

28件のテストケース:

1. **インターフェースパススルー**: GenrePlugin インターフェースが spawnDensity を保持している
2. **JSONGenrePlugin パススルー**: JSON 経由で spawnDensity が正しく伝播される
3. **フォールバック動作**: spawnDensity が未設定の場合、HAZARD_SPAWN が使用される
4. **計算式の検証**: `baseInterval * exp(-decayRate * distance)` の計算が正しい
5. **minInterval クランプ**: 計算結果が minInterval 未満にならない
6. **mergeSpawnDensity**: GameRegistry への統合が動作する
7. **resetRegistry** (テスト用): 開発/テスト時のみ有効

## 使用方法

### 新ジャンルに spawnDensity を追加する場合

1. `src/data/genres/my_genre.json` に `spawnDensity` を追加:
   ```jsonc
   {
     "id": "my_genre",
     "label": "マイジャンル",
     "spawnDensity": {
       "baseInterval": 2000,
       "minInterval": 800,
       "decayRate": 0.00015
     }
   }
   ```

2. それ以外のコード変更は不要。`src/genres/index.ts` が自動的に処理する。

### 既存ジャンルの spawnDensity を調整する場合

対応する `src/data/genres/{genre}.json` の `spawnDensity` を編集するだけ。再ビルドが必要。

## 制限・KnownIssues

1. `mergeSpawnDensity` は `import.meta.env.PROD` 判定に依存しない（いつでも呼べる）。ただし `resetRegistry()` は PROD では警告を出す。
2. `as unknown as` カストは型安全性を妥折しているが、初期化の 1 回限りに限定されている。
3. spawnDensity が JSON に定義されても、`GAME_CONFIG.genres.genres` に含まれていれば自動的に適用される。
