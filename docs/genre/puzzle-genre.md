# パズルジャンル実装ドキュメント

## 概要

MANUAL-OVERRIDEゲームの「パズル（puzzle）」ジャンルを **スライド移動パズル** として実装しました。
プレイヤーが説明書の選択肢を積み重ねることで、横スクロールゲームから思考型のスライドパズルへと変貌します。

グリッド上のプレイヤー駒を上下左右に滑らせ、壁・障害物・盤面の縁にぶつかるまで一直線に移動させて、
ゴールマスへ「ぴたりと止める」ことを目指すパズルです。出題は無限に続き、時間切れで残機を失い、
残機が尽きるとゲームオーバーになります。

## アーキテクチャ

### ファイル構成

```
src/
├── data/
│   ├── config/
│   │   ├── genres.json          # themeColors の puzzle 定義（genres 配列は未使用 / 後述）
│   │   └── puzzle.json          # スライドパズルのパラメータ（サイズ・時間・生成・時間逓減）
│   └── genres/
│       └── puzzle.json          # puzzle ジャンル定義（収束閾値・features・スコア式・操作）
├── genres/
│   ├── PuzzlePlugin.ts          # 視覚テーマ（白系・方眼紙グリッド罫線）
│   └── index.ts                 # プラグイン自動登録（default export を自動収集）
├── game/
│   ├── systems/
│   │   ├── PuzzleFeature.ts     # ゲームロジック（スライド・盤面生成・描画・演出）
│   │   └── index.ts             # フィーチャー登録
│   └── sideScroller.ts          # lights_out 有効時はジャンプ・横スクロール本体の移動を抑止（noControlMode）
├── framework/config-types.ts    # PuzzleConfig / PuzzleGridConfig 型定義
└── domain/types.ts              # ManualTheme 型に 'puzzle' を含む
```

> **注**: ジャンル定義の本体（閾値・features・スコア式など）は `src/data/genres/puzzle.json` から
> 自動収集されます（[`src/data/config.ts`](../../src/data/config.ts)）。`src/data/config/genres.json` は
> **themeColors のみ**が使われ、その `genres` 配列は読み込まれません。両者の puzzle 閾値が食い違っている
> 場合（例: `config/genres.json` 側の `combo:16`）、**実効値は `data/genres/puzzle.json` 側**です。

### フィーチャーIDについて

スライドパズルのゲームロジックは、既存のフィーチャーID **`lights_out`** に実装されています。
これは旧実装（LightsOut パズル）から流用したIDで、`genres.json`・`sideScroller.ts`・テスト等への
波及変更を最小限に抑えるため、ID名はそのままにゲーム内容のみをスライドパズルへ差し替えています。
（クラス名・状態・描画はすべてスライドパズル仕様です。）

## ジャンル収束条件

| パラメータ | 閾値 |
|-----------|------|
| `combo`   | 5以上 |

`combo` 軸が閾値を超えるとパズルジャンルへ収束します（単一軸収束）。

### 収束のしくみ

`combo` は説明書カードの選択で加算されます（`genreParams.combo` による軸加算、または
`genrePoints.puzzle` による直接加点）。収束判定は3方式併用で、主方式の **ベイズ収束**
（`bayes.json`）により `combo` 軸が支配的になった時点で `puzzle` へ確定します。
「正解を積み重ねる／落ち着いて考える」系の選択が `combo` を押し上げる導線になっています。

## ゲーム仕様

### グリッドとサイズ難易度

盤面は正方グリッドで、最小 4×4〜最大 8×8 の5段階。出題が進む（`puzzleCount` が増える）ほど
大きいサイズの出題比率が上がるよう、既存 spawnTable と同じ重み線形補間
（`t = min(1, puzzleCount / weightMaxDist)`）でサイズを抽選します。

| サイズ | 目標最短手数 | 制限時間（基準） | weightStart | weightEnd |
|--------|--------------|------------------|-------------|-----------|
| 4×4    | 2〜3手       | 15秒             | 8           | 2         |
| 5×5    | 3〜5手       | 18秒             | 4           | 4         |
| 6×6    | 4〜6手       | 22秒             | 2           | 5         |
| 7×7    | 5〜8手       | 26秒             | 0           | 4         |
| 8×8    | 6〜10手      | 30秒             | 0           | 3         |

数値はすべて [`src/data/config/puzzle.json`](../../src/data/config/puzzle.json) に定義され、コードには直書きしません。

### 盤面要素

| 要素 | 説明 |
|------|------|
| プレイヤー駒 | スライドする操作対象。立体タイル＋光沢＋中央スタッドの意匠 |
| ゴールマス | 緑のターゲット。ここに **止まれば** 正解（通過しただけでは不可） |
| 壁／障害物 | スライドを止める立体ブロック |
| 空きマス | 通過可能なセル |

### スライドのルール

- 方向キーを押すと、その方向へ **壁・障害物・盤面の縁にぶつかるまで一直線に滑り続ける**（1マス移動ではない）。
- 滑り終えた停止位置がゴールマスと一致したら正解。ゴールはストッパーではないため、止まる位置を計算して導く必要がある。

### 盤面生成アルゴリズム（方式C：目標手数つき生成）

1. グリッド内にランダムに壁を配置する（壁密度 `wallRatio = 0.18`）。
2. 壁でないセルからスタート位置・ゴール位置をランダムに選ぶ（重複なし）。
3. BFS で「スライド移動」基準の最短手数を計算する。**探索の1手＝「ある方向に壁/障害物/縁へぶつかるまで直進」**。
4. 最短手数が当該サイズの目標手数範囲に収まれば採用する。
5. 範囲外（解なし・短すぎ・長すぎ）なら盤面を破棄して 1〜4 を再試行する。
6. 再試行は `maxGenAttempts = 400` 回を上限とし、到達時は **目標範囲に最も近い手数の盤面へフォールバック**（無限ループ防止／解ありを優先）。

BFS の状態空間は停止位置（最大 n² ≦ 64）に限られるため、8×8 でも 1 盤面あたり概ね 0.15ms 程度で生成できます
（壁密度 0.18・上限 400 回での実測。全サイズで採用率ほぼ 100%）。

### 操作方法

| キー | 動作 |
|------|------|
| ← →  | プレイヤー駒を左右にスライド |
| ↑ ↓  | プレイヤー駒を上下にスライド |
| Space | 現在の問題を初期配置にリセット（残機減少なし・回数無制限） |

`Space` は壁配置で身動きが取れなくなった場合の救済措置です。盤面・タイマー・残機は維持したまま、
駒だけを初期位置へ戻します。`Space` は内部的には `controls.jump` を参照しており、パズル中は
横スクロール本体のジャンプ検出が抑止されているためジャンプには化けません。

### 制限時間の逓減（出題数グラデーション）

出題が進むほど各問題の制限時間を **滑らかに（非線形・指数）逓減** させます。

```
scale = max(timeScaleMin, 0.5 ^ (puzzleCount / timeHalfLifeSteps))
制限時間 = サイズの基準時間 × scale
```

既定パラメータ（`puzzle.json`）: `timeScaleMin = 0.25`, `timeHalfLifeSteps = 49`。

| 問 | 制限時間（基準比） |
|----|--------------------|
| 第1問   | 100%（元の時間） |
| 第25問  | 約71% |
| 第50問  | **丁度 50%** |
| 第75問  | 約35% |
| 第99問  | 25%（下げ止まり到達） |
| 第100問以降 | 25%（据え置き） |

`timeHalfLifeSteps` 問ごとに半減し、`timeScaleMin` で下げ止まります。値はすべて JSON 化されています。

### 残機

- `hp` フィーチャー（[`RpgFeature`](../../src/game/systems/RpgFeature.ts)）を流用します。
- 初期残機は3（`physics.json` の `defaultPlayerMaxHp: 3`）。
- 残機は画面下のハート（♥/♡）で表示。

### スコア

最終スコア式（`src/data/genres/puzzle.json`）:

```
combo * 200 + survivedSec * 3
```

- 正解（ゴール到達）でコンボ +1。
- 時間切れでコンボはリセット（`world.resetCombo()`）し、残機 −1。
- パズル中は `scrollSpeed = 0` のため `distance` 項は使用しません。

### ゲームフロー

```
問題出現（scrollSpeed = 0 でスクロール停止）
  ↓
ゴール到達（正解）  → コンボ+1 → CLEAR! 演出（軽いシェイク）→ 即・次の問題
時間切れ           → 残機-1   → TIME UP 演出（強めのシェイク）→ 即・次の問題
残機0              → ゲームオーバー
```

- 問題と問題の間にスクロール再開区間は設けません。
- 切り替え時は既存のスクリーンシェイク・スコアポップアップ・パーティクルで演出します。

## ビジュアル

- **背景（方眼紙テーマ）**: パズル中は全画面を白系（`#f0f0f0`）＋薄いグリッド罫線（`opacity 0.1`、数セルごとに太罫）で
  描き直します。これにより横スクロール本体（地面・base プレイヤー）を覆い隠しつつ、puzzle テーマの「方眼紙」感を出します。
  - 旧実装では暗いオーバーレイ（`rgba(8,8,26,0.9)`）が白背景を覆って紺色に見える不具合がありました。現在は撤廃済みです。
- **盤面パネル**: 影付きの白カード＋二重枠＋四隅オーナメント（脈動）で盤面外を装飾。
- **プレイヤー駒**: 縦グラデの立体タイル、輪郭、上側ハイライト、中央スタッド（微脈動）で視認性を強化。
- **ゴール**: 緑の同心円ターゲット（脈動）。
- **スライドアニメ**: `easeOut` 補間。所要時間は移動距離に比例（1セルあたり `slidePerCellSec`、最短 `slideMinSec`）。
- **演出**: 正解＝緑フラッシュ＋「CLEAR!」、時間切れ＝赤フラッシュ＋「TIME UP」。いずれも縁取りテキスト＋パーティクル。

`PuzzlePlugin.ts` 側でも genre テーマ（白系 skyColors・方眼紙 `drawMidLayer`）を整備しており、
ジャンル確定演出など overlay 表示前の瞬間も白系で一貫します。

## 設定パラメータ（puzzle.json）

| キー | 既定値 | 内容 |
|------|--------|------|
| `grids[]` | 5段階 | サイズ別の `n` / `minMoves` / `maxMoves` / `timeSec` / `weightStart` / `weightEnd` |
| `cellPx` | 48 | 1セルの描画ピクセル |
| `weightMaxDist` | 12 | サイズ重み補間が最大化するまでの出題数 |
| `wallRatio` | 0.18 | 各セルが壁になる確率 |
| `maxGenAttempts` | 400 | 盤面生成の再試行上限（到達時は最近傍へフォールバック） |
| `timeScaleMin` | 0.25 | 制限時間スケールの下限 |
| `timeHalfLifeSteps` | 49 | 制限時間が半減するまでの問題数 |

## テスト

### 静的検証 (`tests/feature-lights_out.test.mjs`)

ブラウザ/開発サーバーなしで判定できるよう、`PuzzleFeature.ts` の実装内容と関連 config の整合性を
静的に検証します。`handles=['lights_out']` の存在、`scrollSpeed=0` / `baseScrollSpeed` 復元、
`_handleSolved` / `puzzleCount++` / `_startPuzzle` / `setCombo`、クリア/ダメージ演出（`solveFx`/`damageFx`/
`addParticle`/`triggerShake`/`soundManager`）、操作抑止（`player.vx=0`/`player.vy=0`）、
`genres` 側で `lights_out` が有効化されていること等を確認します（**PASS**）。

### 生成アルゴリズムのベンチ（補助）

方式C生成の妥当性・速度を別スクリプトで再現検証し、全サイズで採用率ほぼ100%・8×8（6〜10手）で
約0.15ms/盤面・最大試行数 < 上限 を確認しています。

### E2E (`tests/puzzle-advance.spec.ts`)

矢印キーでプレイヤー駒をゴールまで滑らせ、`solvedCount` / `puzzleCount` が進むことを実機で検証します。
解法は「壁/境界にぶつかるまで直進」を1手とした BFS（`_generateBoard` と同じスライド規則）で求め、
`window.__puzzleState`（DEBUG 補助）から盤面・進捗を読み取ります。

## 実装上の注意点

1. `preUpdate` / `update` でプレイヤー速度を 0 に固定（`vx=0, vy=0`）し、横スクロールの慣性とハザードを抑止する。
2. `onInit` で `scrollSpeed` を `baseScrollSpeed` に退避して `0` に設定し、`onDisable` で復元する。
3. ジャンル確定時にエンジンが呼ぶのは `onManualUpdated` のみ（`onInit` は実行時には呼ばれない）ため、
   `onManualUpdated` から `onInit` を呼んで初期化する。
4. 全メソッド（`preUpdate`/`update`/`render`）の冒頭で `features.has('lights_out')` ガードを行う。
5. 方向キー名・リセットキー名は `onInit` で `world.rules.controls` からキャッシュする
   （実行中に変化しないため毎フレーム参照しない）。
6. 盤面生成・スライド最短手数・時間スケールはモジュールレベルの純粋関数
   （`_generateBoard` / `_bfsMinMoves` / `_slideDest` / `_timeScale`）として実装。
7. 視覚・寸法・演出の定数はすべて `readonly` フィールド、またはゲーム値は `puzzle.json` に集約（マジックナンバー排除）。
8. 開発ビルドでのみ `window.__puzzleState` にパズル状態を公開（動作確認用、本番では除去）。
9. ゴール判定は **停止位置がゴールに一致した時のみ**（通過は不可）。スライドアニメ完了時に判定する。

## 実装経緯

1. **初版（スライドパズル化）**: 旧 LightsOut 実装を全面置換し、スライド移動パズル＋方式C生成（BFS）＋
   キーボード操作＋クリア/時間切れ演出を実装。フィーチャーIDは `lights_out` を流用。
2. **仕様更新**: スペースキーによる初期配置リセット（救済措置）を追加。目標最短手数の上限を10手まで引き上げ
   （5×5=3-5 / 6×6=4-6 / 7×7=5-8 / 8×8=6-10）。プレイヤー駒・盤面外装飾の作り込みと、
   背景が紺色になる不具合（暗オーバーレイが原因）の修正。
3. **時間逓減**: 出題数に応じて制限時間を半減期型で逓減（第50問=50%、第100問=25%で下げ止まり）。
4. **E2E更新**: `puzzle-advance.spec.ts` を旧 LightsOut（セルクリック）前提から矢印キー操作のスライド版へ更新。
