# {ジャンル名}ジャンル実装ドキュメント フォーマット

`docs/genre/` 配下の各ファイルは、1ジャンルの実装を記録するドキュメントです。
新しいジャンルを実装・改修したら、このテンプレートに沿って `docs/genre/{genre-id}-genre.md` を作成・更新してください。
（参考実装: [tetris-genre.md](tetris-genre.md), [puzzle-genre.md](puzzle-genre.md)）

セクションは原則この順序で記載します。「(任意)」と付いたセクションは、ジャンルの実装内容に応じて省略可です。

---

```markdown
# {ジャンル名}ジャンル実装ドキュメント

## 概要

何を実装したか（1〜3文）。横スクロールからどう変貌するか、プレイヤー体験の核を簡潔に。

## アーキテクチャ

### ファイル構成

新規・変更したファイルをツリーで列挙し、各行に役割をコメントで添える。

```
src/
├── data/
│   ├── config/{genre}.json     # ゲームパラメータ
│   └── genres/{genre}.json     # ジャンル定義（収束閾値・features・スコア式）
├── genres/
│   └── {Genre}Plugin.ts        # 視覚テーマ
└── game/systems/
    └── {Genre}Feature.ts       # ゲームロジック
```

既存フィーチャーIDを流用する等、命名と実体が食い違う場合はここで注記する（例: puzzle-genre.md の「フィーチャーIDについて」）。

## ジャンル収束条件

| パラメータ | 閾値 |
|-----------|------|
| `axis`    | N以上 |

どの軸がどう収束に効くかを表で示す。複数軸が必要なら AND/OR を明記する。

### 収束のしくみ (任意)

genreParams / genrePoints / ベイズ収束のどの方式で収束させているか。
分岐パスの到達検証を行った場合はその結果も記載する。

## ゲーム仕様

ジャンル固有のルール・データ構造（グリッド、敵パターン、判定条件等）をサブセクションで詳述する。

### 操作方法

| キー | 動作 |
|------|------|
| ← → | ... |

### スコア計算

最終スコア式と、各項の意味・特記事項（例: scrollSpeed=0 のため distance 項が常に0 になる等）。

## 説明書分岐 (任意)

分岐JSON（manuals/ や cards/）のエントリ数・バージョン範囲、既存分岐木との接続ポイントを記載する。

## ビジュアル (任意)

背景・UI・演出など見た目に関わる変更点。

## 設定パラメータ ({genre}.json) (任意)

| キー | 既定値 | 内容 |
|------|--------|------|
| ... | ... | ... |

config JSON のキーごとの意味を一覧化する。

## テスト

### ユニットテスト (`tests/...`)

テスト対象・件数・結果（PASS/FAIL件数）。

### Playwright E2Eテスト (`tests/...`) (任意)

確認しているシナリオを箇条書きで。

## 実装上の注意点

実装の非自明な制約・回避策を番号付きリストで列挙する（CLAUDE.md のコメント方針に準拠し、
「なぜそうしているか」を書く。「何をしているか」だけの説明は不要）。

## レビュー履歴 (任意)

コードレビューを反復した場合、Iteration ごとに HIGH/MEDIUM/LOW件数と主要な修正内容を記録する。

## 実装経緯 (任意)

複数回の仕様変更を経た場合、時系列で要約する。

## 今後の改善候補

未実装・既知の妥協点を箇条書きで。
```

---

## 運用ルール

- ファイル名は `docs/genre/{genre-id}-genre.md`（`genre-id` は `src/domain/types.ts` の `GenreId` と一致させる）。
- 数値・閾値・スコア式は、実装（JSON/コード）と乖離していないか必ず確認してから記載する。乖離がある場合は
  「どちらが実効値か」を明記する（puzzle-genre.md の `config/genres.json` vs `data/genres/puzzle.json` の注記を参照）。
- 既存フィーチャーID・クラス名を流用した場合は、ドキュメント内で必ずその旨を明記する（コードを読むだけでは
  「なぜこの名前なのか」が分からないため）。
- レビュー履歴は HIGH/MEDIUM/LOW の指摘内容と対応を残し、再発防止の参考にできるようにする。
