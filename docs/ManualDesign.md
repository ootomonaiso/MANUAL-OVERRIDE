# 説明書の仕様についての改善案

このファイルは、kissi-710が説明書の仕様改善と改善案を訴えるためのドキュメントです。

対象は「説明書（取扱説明書）」の**データ構造と分岐の仕組み**に限定する。
本ファイルは説明書の設計に絞って掘り下げる。

---

## 現在の仕様（現状把握）

### データ構造

- 説明書は `MANUAL_DECK`（`src/data/manualDeck.ts`）という、**version文字列をキーにしたフラットな辞書**として保持される。
- 各エントリは `ManualVersion`（`src/domain/types.ts:170`）で、`manualText` / `controls` / `hazards` /
  `choices[]` などを持つ。
- 分岐は各 `choices[].next`（`src/domain/types.ts:142`）が**次のキー文字列を直接指す**ことで表現される。
  つまり全体は「文字列キーで結ばれた木（実体は DAG）」になっている。
- ノードの実体は4つの JSON に分割定義されている。
  - `base.json` … ルート `1.0` と最初の2択（`2.0-a` / `2.0-b`）
  - `action-branch.json` … `2.0-a` 系（戦闘・探索方向、〜`8.0-a-*` / `6.0-a-*`）
  - `flow-branch.json` … `2.0-b` 系（速度・リズム方向、〜`8.0-b-*`）
  - `advanced-branch.json` … `9.0`〜`15.0` 系
- `import.meta.glob` で `manuals/*.json` を自動収集するため、**ファイルを置けばデッキには載る**。
  ただし「デッキに載る」ことと「プレイ中に到達できる」ことは別問題（後述）。

### 分岐・ジャンル収束の仕組み

- 選択するたびに `choice.genreParams`（`tempo` / `range` / `enemy` / `combo` ほか全12軸）が累積される
  （`accumulateParams`, `src/domain/genreResolver.ts:6`）。
- 累積値が `GENRES` の各 `thresholds` を**全て満たした**ジャンルが「収束候補」になり、超過量が最大の
  ものが選ばれる（`resolveGenre`, `genreResolver.ts:24`）。
- 確定タイミングは `useGameState.choose`（`src/composables/useGameState.ts:78`）。
  `nextVer.choices.length === 0`（末端）**または** `resolved !== 'base'`（いずれかのジャンルが閾値到達）
  になった瞬間に `genreLocked` へ遷移する。

## 問題点

### 1. 説明書がツリー形状で、選択肢を内包している（単体追加が困難）

説明書ノードが次の選択肢（`next`）を内部に持つツリー構造のため、説明書を追加する場合、その枝ごと
追加する必要に駆られる。説明書を単体で追加することが考えられていない。

**裏付け**: 新ノードを実際にプレイへ繋ぐには、最低でも
①親ノードの `choices[].next` を新ノードのキーに向ける、②新ノード自身の `next`（または末端なら空 `choices`）
を用意する、の両方が要る。`MANUAL_DECK` に置くだけでは `validateManualVersion` の到達性チェックで
「どこからも参照されていない（到達不可能）」警告になる（`src/framework/ManualValidator.ts:118-134`）。
1枚を足すのに前後の配線を必ず触る＝単体での独立追加ができない。

### 2. ジャンル追加時の不便さ

ジャンルを追加した場合、そのジャンルが実際に到達できるよう、説明書を用意する必要が出てくる。
その時、ツリー全体とそのジャンルのしきい値を見て枝ごと生成する必要がある。

**裏付け（さらに踏み込んだ事情）**: ジャンルは閾値の AND 条件（`thresholds` 全軸クリア）で確定するが、
確定は「**いずれかのジャンルが最初に閾値到達した時点で即発火**」する（`useGameState.ts:78`）。
そのため新ジャンルを到達可能にするには、単に閾値を満たすパスを作るだけでは足りず、
**そのパスの途中で他の（より低閾値の）ジャンルに先に到達しないよう**、経路上の `genreParams` の積み上がり方
まで設計しなければならない。既存の低閾値ジャンル（`stealth_action {stealth:4}` / `puzzle {combo:4}` /
`idle {craft:4}` / `runner {tempo:5}` 等、`src/domain/types.ts:31-51`）が「先に成立」しやすく、追加ジャンルが
食われやすい。ツリー全体の累積を読み切る必要があり、これが「枝ごと生成」の負担を増している。

### 3.（実害として現れている例）9.0〜15.0 の説明書が丸ごと到達不能

問題1・2が実際に表面化している証拠として、`advanced-branch.json` の `9.0`〜`15.0` ノード群が
**現状まったく到達できない**。

- **グラフ的に孤立**: `9.0-a`〜`9.0-d` を `next` に指すノードがどこにも存在しない
  （`grep '"next":"9..."' src/data/manuals/` が 0 件）。`8.0-a-*` / `8.0-b-*` は `choices: []` の末端で、
  そこから 9.0 へ繋がっていない。
- **プレイ的にも到達不能**: 仮に配線しても、8.0 に着く前に問題2の早期確定で `genreLocked` 済み。
  Phase B/C を抜けて 9.0 以降へ進む経路が機能していない。
- dev バリデータはこれを「到達不可能」警告として出すはずだが（`ManualValidator.ts:130-134`）、
  **警告どまりで本番ビルドでは検証自体スキップ**（`ManualValidator.ts:176` の `if (import.meta.env?.PROD) return`）。
  孤立コンテンツがビルドを止めずに出荷される。

> CLAUDE.md は「ver 9.0〜15.0、無限選択肢システム（100+ 選択肢）」を実装完了と記載しているが、
> 現状の配線・確定ロジックではこれらが遊べない。仕様意図とデータの乖離。

### 4. 「いずれか即確定」ルールが、分岐の深さと矛盾している（推測を含む）

問題2・3の根本。`choose()` の確定条件 `resolved !== 'base'` は「最初に閾値を満たしたら止める」挙動で、
深い分岐ツリー（最大15段）を用意しても、実際には序盤（4〜6段目あたり）で確定しがち。

- これが**意図的設計**（早めに個性を固定したい）である可能性は否定できない。ただしその場合、
  9.0+ の深いツリーを用意していることと整合しない。
- 仕様意図がどちらなのか不明なため、ここは**推測**として記録する。少なくとも「深いツリー資産」と
  「早期確定ロジック」は現状かみ合っていない。

## 改善案

ここからは、今のところ思ってることを書きます。後で変わるかもしれないし、もっといい案もあるかもしれないけど。

### 案1.　バージョンごとに出る説明書をまとめてしまおう

#### 概要

バージョンごとに、[AAAA.json, BBBB.json, ...] というリストを作成して、そこからランダムに表示されるようにしよう。
ジャンルの追加があっても、取る側が工夫して取ればどんなジャンルでも到達できる。

分かりづらいので例:

- バージョン1:[AAAA.json, BBBB.json, CCCC.json]
- バージョン2:[DDDD.json, EEEE.json, FFFF.json]
- バージョン3:[GGGG.json, HHHH.json, KKKK.json]

このとき、AAAA → FFFF → KKKK というふうに取ったり、BBBB → DDDD → GGGG というふうに取れば、各バージョンの中からどれを選ぶかの組み合わせで、どのようなジャンルでも到達できる。
枝を1本1本つなぐ代わりに、「何バージョン目のプールに入れるか」だけ決めればいい。

#### 何が嬉しい？

説明書を1枚足したいときは、対象バージョンのリストに名前を1行足すだけで済む。前後の `next` を配線する必要がなくなる

ジャンルを足すときも、そのジャンルに効く説明書を各バージョンのプールに何枚か撒いておけば、取り方次第で到達できる経路が勝手に生まれる

ただ、バージョンという段階自体は残る。

### 案2.　いっそのこと「説明書の分岐」をやめてしまおう(本命)

#### 概要

プレイヤーには、現在プレイしているジャンルと、今積んであるジャンルパラメータの数値、説明書を選んだことによって得られるパラメータ(或いは減るパラメータ)を見ることができる。
説明書だけを大量に実装し、中にパラメータを入れる。

説明書はランダム(或いはなにかしら重みをつけた確率)で選択肢に用意される。
選択すると、パラメータが増えて、しきい値を超えた中で一番超過の多いジャンルが決定される。

バージョンという段階すら持たず、「全説明書プールから毎回ランダムに数枚出す」だけ、というところまで振り切る。

#### 何が嬉しい？

説明書同士どころか、ジャンルまでもすべて切り離される。説明書の単体追加が容易となり、大量のコンテンツ追加が期待できる。
また、現在はいつも見る選択肢しかないが、ランダムに選ぶ仕様に変更することで、大量の説明書群の中から組み合わせて表示されるため、プレイヤーは慣れが来ないはず。
レアな説明書、普遍的な説明書、みたいに説明書ごとに確率をいじってしまうのも面白いかもしれない。

おまけで、パラメータの増減と現在地をプレイヤーに見せる方針は、「genreParamsの蓄積が見えない」問題も一緒に片付く。
確定タイミングも「しきい値を超えた瞬間」ではなく「更新回数を使い切ったら最終判定」に寄せれば、早期確定で分岐が消化されない問題

## 99. 少し脱線した改善案

書いてる途中に、「直前に取った説明書が特定ジャンル」なら、このセリフ、とかもできないこともないな、と思った。

あとは、説明書を選ぶときに、これを選んだら〇〇確定！！みたいなのもいいなと思った。すでに見たジャンルとかを何処かで持っておいて、未確認なら???ジャンル確定、確認済みなら{ジャンル名}確定とか、もしくは何かしらが確定するとかで誤魔化してもいいし。

今はどの方向のジャンルに進んでいるみたいなのもあるとわかりやすいかなって思った。


## 補足（確証レベルの整理）

- **事実（コード・grepで確認）**: ツリー構造で枝が内包されること（問題1）、早期確定ロジックの存在（問題2）、
  9.0+ ノードがどの `next` からも参照されていないこと、バリデータが本番でスキップされること（問題3）。
- **推測**: 早期確定が意図的設計かどうか（問題4）。案1・案2の効果は設計上の見込みであり、
  実装・計測で検証が必要。
